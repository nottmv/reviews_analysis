# Финальный проект: Маркетинговый аналитик по отзывам на музыкальные приложения

## 1. Задача и данные

**Задача:** создать вопросно-ответную систему для анализа отзывов на музыкальные стриминговые сервисы. Система должна по запросу в свободной форме находить релевантные отзывы, извлекать из них аспекты с цитатами и формировать ответ, проверенный на галлюцинации.

**Данные:** реальные отзывы пользователей из Google Play на три приложения - «Звук» (Сбер), «Яндекс Музыка» и «ВК Музыка». По 200 отзывов на каждое приложение, собранные парсингом. Общий объём - 600 отзывов. Формат CSV (колонки: `rating`, `content`, `userName`, `at`, `reviewCreatedVersion`).

## 2. Система

В проекте используются четыре техники курса, объединённые в единый пайплайн:

1. **RAG (Retrieval-Augmented Generation):** индексация всех отзывов через `SentenceTransformer` (`paraphrase-multilingual-MiniLM-L12-v2`) и поиск top‑5 релевантных отзывов.
2. **Information Extraction + аспектный анализ:** извлечение из найденных отзывов структурированных аспектов (`sound_quality`, `ui_design`, `price`, `ads`, `recommendations`, `stability`, `speed`, `support`) с указанием тональности, цитаты и уверенности.
3. **LLM‑as‑Judge:** независимая проверка ответа на соответствие цитатам и отсутствие галлюцинаций.
4. **Агент с инструментом:** для вопросов сравнения вызывается `compare_products`, который собирает отзывы по двум продуктам и генерирует сравнение.

Все вызовы LLM используют `make_client()` с `max_retries=3` и `response_model`. Присутствует `field_validator` (рейтинг 1-5, дата не позже сегодня). Галлюцинации проверяются встроенным в судью механизмом.

**Схема пути данных:**
1. Загрузка CSV с отзывами.
2. Индексация через `SentenceTransformer`.
3. Приём вопроса -> retrieval top‑5.
4. Извлечение аспектов из объединённого текста отзывов (batch‑вызов LLM).
5. Если вопрос содержит «сравни»/«vs» -> вызов агента `compare_products`.
6. Сборка финального ответа с цитатами.
7. Проверка судьёй и возврат результата.

## 3. Оценка

Eval проведён на 15 вопросах разных типов (прямые, синонимы, сравнения, multi‑hop, позитивные, трудные). Метрики (см. `output/eval_results.json` и `output/summary.json`):

- **Judge Pass Rate:** 10/15 (67%).
- **Groundedness:** 15/15 (100%) - все ответы подкреплены цитатами.
- **Среднее время ответа:** 23.1 сек.

**Путь и стоимость:**
Каждый запрос проходит через строго определённое число шагов, фиксируемое в результатах:
- 3 шага для обычных вопросов: RAG‑retrieval, batch‑извлечение аспектов (IE) и LLM‑judge.
- 4 шага для вопросов сравнения: дополнительно вызывается инструмент `compare_products`.

Используемые инструменты отражены в поле `tools_used` для каждого вопроса (см. `eval_results.json`).  
Среднее число LLM‑вызовов на вопрос - 3.2. Оценка затрат: при длине запроса ~2000 токенов и ответа ~200 токенов каждый вызов потреблял около 2200 токенов. За 15 вопросов израсходовано примерно 100 тыс. входных и 12 тыс. выходных токенов. При стоимости API DeepSeek-V4 Flash ($0.14 / $0.28 за 1M токенов) полный прогон eval обошёлся бы примерно в $0.013 (около 1.2 руб).

## 4. Где пайплайн ошибался

**Ошибка 1 (Q9): не выполнено сравнение**
Вопрос: «Сравни удобство интерфейса Яндекс Музыки и ВК Музыки». Ответ содержал только общие цитаты, без указания, к какому сервису они относятся. Судья вернул `ok=false` с причиной: «Ответ не содержит сравнения двух сервисов». Это системная проблема: batch-извлечение аспектов не привязывает цитаты к конкретному продукту, что критично для сравнительных вопросов.

**Ошибка 2 (Q13): подмена тональности**
Вопрос: «Какие сильные стороны отмечают у Яндекс Музыки?». Ответ содержал только негативные аспекты (цена, рекомендации), хотя запрашивались позитивные. Судья указал: «Ответ включает негативные отзывы, хотя вопрос о сильных сторонах». Причина: модель не фильтрует аспекты по тональности, ориентируясь на общий контекст отзывов, где преобладает негатив.

**Ошибка 3 (Q10): неполное покрытие multi‑hop**
Вопрос: «Какие пожелания пользователей встречаются одновременно и у Звука, и у Яндекс Музыки?». Ответ перечислил несколько аспектов, но не выделил действительно общие для двух сервисов. Судья отметил: «Не указаны пожелания, общие для Звука и Яндекс Музыки». Пайплайн пока не умеет агрегировать информацию из разных источников и находить пересечения.

## 5. Выводы

Система успешно отвечает на фактологические вопросы о музыкальных приложениях, подтверждая каждый ответ цитатами из реальных отзывов (100% ответов имеют цитаты). 67% ответов одобрены судьёй — это демонстрирует практическую применимость для первичного маркетингового анализа.

**Границы применимости:** система плохо справляется с задачами, требующими синтеза идей (Q15), строгого сравнения (Q9) и выделения общих паттернов (Q10). Точность зависит от качества retrieval: если в top‑5 не попали нужные отзывы, ответ будет неполным.

**Для production:**
- Оставили бы RAG на SentenceTransformer и judge‑проверку - они работают надёжно.
- Доработали бы классификатор типа вопроса и для синтетических запросов добавили бы творческий LLM‑синтез на основе цитат.
- Для сравнительных вопросов внедрили бы явную привязку цитат к продуктам.
- Добавили бы map‑reduce для multi‑hop запросов, чтобы гарантировать полноту покрытия.
